# 01-故障顺序与判断

## 阶段 1：插件入口消失

现象：

```text
Browser 插件不可用
Chrome 插件不可用
Computer Use 插件不可用
插件市场搜不到 Browser / Chrome / Computer Use
```

判断：

```text
openai-bundled marketplace 没有被 Codex 正确发现。
```

关键路径：

```text
%USERPROFILE%\.codex\.tmp\bundled-marketplaces\openai-bundled
```

应存在：

```text
openai-bundled\.agents\plugins\marketplace.json
openai-bundled\plugins\browser\.codex-plugin\plugin.json
openai-bundled\plugins\chrome\.codex-plugin\plugin.json
openai-bundled\plugins\computer-use\.codex-plugin\plugin.json
```

技术原理：

```text
Browser / Chrome / Computer Use 是 Codex 内置 openai-bundled 插件。
marketplace 缺失时，即使 config.toml 启用了插件，线程也不会加载工具。
```

---

## 阶段 2：Chrome 扩展 Disconnected

现象：

```text
Chrome 扩展显示 Disconnected
```

判断：

```text
Chrome Native Messaging Host 没有生成或注册。
```

关键文件：

```text
%LOCALAPPDATA%\OpenAI\extension\com.openai.codexextension.json
```

关键注册表：

```text
HKCU\Software\Google\Chrome\NativeMessagingHosts\com.openai.codexextension
```

关键字段：

```json
"allowed_origins": [
  "chrome-extension://hehggadaopoacecdllhhajmbjkdcmajg/"
]
```

技术原理：

```text
Chrome 扩展必须通过 Native Messaging Host 调用本地 extension-host。
```

---

## 阶段 3：Chrome Connected 但仍不能打开网页

现象：

```text
Chrome 扩展已 Connected
但 @Chrome / @Browser 仍不能打开网页
Computer Use 仍不可用
```

判断：

```text
Chrome 扩展链路已通，但 Codex 运行时执行入口缺失。
不要继续重装 Chrome 扩展。
```

关键日志：

```text
mcp__node_repl__js
node-repl-missing
missingHelperPath=true
browser_use_setup_failed
```

技术原理：

```text
Connected 只代表 Chrome 扩展与 Native Host 连通。
真正控制网页还需要 node_repl、cua_node、codex.exe、rg.exe、sandbox helper、command runner。
```

---

## 阶段 4：WindowsApps 迁移失败

现象：

```text
bundled_executable_relocation_failed
普通复制失败
错误：无法加密指定的文件
错误 6000
```

判断：

```text
WindowsApps 中的 MSIX 文件带特殊保护或加密属性。
普通复制会失败。
```

正确方向：

```text
用文件流复制，只复制文件内容，不复制 WindowsApps 的特殊属性。
```

---

## 阶段 5：复制到 bin 根目录仍不够

曾经补齐：

```text
%LOCALAPPDATA%\OpenAI\Codex\bin\codex.exe
%LOCALAPPDATA%\OpenAI\Codex\bin\node.exe
%LOCALAPPDATA%\OpenAI\Codex\bin\node_repl.exe
%LOCALAPPDATA%\OpenAI\Codex\bin\rg.exe
%LOCALAPPDATA%\OpenAI\Codex\bin\cua_node
```

但仍失败。

原因：

```text
Codex 实际读取 fingerprint 版本化目录，不一定读取 bin 根目录。
```

## 阶段 6：按版本化目录修复

本次有效路径：

```text
bin\38dff8711e296435\codex.exe
bin\ada252862d154cdd\rg.exe
runtimes\cua_node\1b23c930bdf84ed6\bin\node.exe
runtimes\cua_node\1b23c930bdf84ed6\bin\node_repl.exe
```

判断：

```text
必须从最新日志读取 CodexFingerprint / RgFingerprint / CuaFingerprint / RuntimePaths。
不要复用旧 hash。
```

## 阶段 7：CODEX_CLI_PATH 与 nodeRepl.config

现象：

```text
所有网站都被 Codex 策略阻止
但设置中是“始终允许”
网站权限列表为空
```

判断：

```text
不是网站黑名单，而是 nodeRepl.config 权限桥失败。
```

处理：

```text
设置 CODEX_CLI_PATH 指向版本化目录中的 codex.exe。
```

## 阶段 8：sandbox / command-runner 缺失

现象：

```text
node_repl 能启动到下一阶段
但提示缺少 codex-windows-sandbox-setup.exe
或缺少 codex-command-runner.exe
```

判断：

```text
不是回到原点，而是同一运行时迁移失败链上的下一层 helper 缺失。
```

处理：

```text
只补缺失 helper。
不要回头重修 Chrome 扩展、Native Host、openai-bundled 或 node_repl。
```

## 阶段 10：旧线程仍不注入 node_repl

现象：

```text
Browser / Chrome / Computer Use 插件均已安装启用
node_repl.exe、node.exe、codex.exe 都存在
Chrome Native Host v2 manifest 存在
codex doctor 不再提示运行时缺失
但当前线程 tool_search("node_repl js") 仍只返回无关工具
mcp__node_repl__js 仍不可调用
```

判断：

```text
这不是 Chrome 扩展问题，也不是 node_repl.exe 文件缺失。
常见原因是旧线程的工具表没有重新注入，尤其是线程创建于旧 CLI / 旧插件状态。
```

检查信号：

```text
state_*.sqlite 的 threads 表中：
cli_version 低于当前 codex --version
history_mode=legacy
source=vscode 或旧来源
thread_dynamic_tools 为空
```

补救方向：

```text
先确认 Browser / Chrome / Computer Use 已安装启用。
再确认 node_repl.exe 可作为 MCP server 启动。
必要时显式添加 node_repl MCP server。
添加后需要新 turn、新线程或重启 Codex 才会刷新工具表。
```
