# 09-2026-07-20 完整修复时间线与文件映射

## 1. 最终结论

本次不是单一插件安装故障，而是同一表象下连续出现了多层问题：

```text
Native Host v2 缺失或注册失效
-> openai-bundled marketplace/cache 半残或版本漂移
-> chrome-native-hosts-v2.json 记录死亡 PID、旧字段或 UTF-8 BOM
-> 旧任务没有重新注入 mcp__node_repl__js
-> 普通外部 node_repl MCP 形成“工具可见但不可信”的假修复
-> Desktop 与 npm CLI 二进制来源混用，可信审批桥缺失
-> WindowsApps EFS 文件复制报 UNKNOWN errno=-4094
-> 按内容哈希定位的 Codex/CUA runtime 未落盘
-> Computer Use helper/transport 缺失，Sky runtime 无法启动
```

最终真正恢复 Computer Use 的关键不是再次重装插件，也不是再次打开权限开关，而是：

1. 按当前 MSIX 文件内容计算 Desktop 和 CUA runtime 的目标哈希目录。
2. 从已校验、与当前 MSIX SHA-256 相同的本地副本预热目标目录。
3. 使用 `CopyFileExW + COPY_FILE_ALLOW_DECRYPTED_DESTINATION` 建立不携带 WindowsApps EFS 属性的 bundled resources 镜像。
4. 将 `CODEX_ELECTRON_BUNDLED_PLUGINS_RESOURCES_PATH` 指向当前 MSIX 版本镜像。
5. 完全退出并手动重启 Codex，新建任务做官方客户端动态验收。

最终验收结果：Chrome 可以通过官方浏览器客户端读取活动标签页；Computer Use 可以通过官方客户端实际执行 `sky.list_apps()` 并返回应用列表。

> 重要修正：当前版本中 `nodeRepl.nativePipe` 可以是 `undefined`，但官方 Computer Use wrapper 仍能通过内部 transport 正常调用 Sky。因此不能再把直接存在 `nodeRepl.nativePipe` 当作唯一成功标准。

## 2. 按发生顺序的修复过程

### 阶段 1：确认工具入口与 Native Host v2

最初现象：任务没有 Chrome/Computer Use 的实际调用入口，Chrome 侧边栏提示：

```text
Codex Chrome native host v2 manifest is missing
```

检查和处理：

- 检查 Chrome/Computer Use skill 是否安装。
- 检查 Chrome Native Messaging manifest 和 HKCU 注册表入口。
- 修复 `%LOCALAPPDATA%\OpenAI\extension\com.openai.codexextension.json` 及浏览器 NativeMessagingHosts 指向。

结果：只恢复了扩展到 Native Host 的第一层连接，尚未恢复任务内可信工具。

### 阶段 2：修复 bundled marketplace、插件缓存和 latest 指向

现象：插件页安装失败、显示正在安装，或插件 UI 显示已启用但任务仍无法调用。

检查和处理：

- 比较当前 MSIX、活动 marketplace、插件缓存和 Native Host 实际引用版本。
- 刷新 `openai-bundled` marketplace。
- 重新安装同版本 Chrome / Computer Use 插件。
- 修复 `latest` junction 和 `.plugin-appserver`。

结果：插件清单层恢复，但仍不能证明可信运行时已经注入。

### 阶段 3：处理早期 runtime helper 缺失

日志出现：

```text
bundled_executable_relocation_failed
node-repl-missing
browser_use_setup_failed
windows sandbox failed: spawn setup refresh
os error 740
node_repl kernel exited unexpectedly
```

早期处理是从当前 MSIX 对照并补齐 `codex.exe`、`node_repl.exe`、sandbox helper、command runner 和 CUA runtime。该处理解决了部分缺文件，但当时仍按显示版本或 manifest tag 推断目标目录，尚未发现 Desktop 实际使用的是内容哈希目录。

### 阶段 4：发现 npm CLI 与 Desktop 二进制不能混用

一次中间修复曾把全局 npm 安装的 `codex.exe` / `codex-code-mode-host.exe` 写入 Desktop bin。即使两边 `codex --version` 相同，SHA-256 仍可能不同。结果是：

```text
mcp__node_repl__js 一度出现
Browser security unavailable outside node repl
随后工具入口再次消失
```

处理：取消 npm 优先策略，Desktop 核心文件只接受当前 MSIX 或与当前 MSIX SHA-256 完全一致的本地副本。

结论：npm 不是 Codex Desktop 的核心依赖；全局 npm 目录不是官方可信运行时来源。

### 阶段 5：旧任务不热加载与外部 MCP 假成功

旧任务不会自动刷新启动时确定的工具表。随后尝试注册普通 stdio `node_repl` MCP，虽然工具入口出现，但报错：

```text
Browser security unavailable outside node repl
Computer Use requires app approval but elicitations are unavailable
```

原因：普通 MCP 没有 Desktop 提供的受信任 `createElicitation`、browser security 和 Computer Use transport 上下文。

处理：移除外部 `node_repl` workaround，后续只在完全重启后的新任务中验收官方工具。

### 阶段 6：权限开关不是审批桥

设置页“任意应用”和 Chrome 开关都已允许，但 Computer Use 仍失败。持久权限开关只记录许可；任务运行时仍需要官方审批/transport 桥。重复切换权限、重装扩展不会补出缺失的 runtime bridge。

### 阶段 7：修复 app-server presence 和 JSON BOM

Chrome 侧边栏后来出现：

```text
No compatible Codex app-server entry was found
Unable to start ChatGPT
expected value at line 1 column 1
```

检查发现 `%LOCALAPPDATA%\OpenAI\Codex\chrome-native-hosts-v2.json` 可能包含死亡 PID、旧字段，或者由 Windows PowerShell 写成 UTF-8 BOM。Rust JSON 解析器在该路径上将 BOM 视为首字符并失败。

处理：

- 完全退出 Codex 和浏览器。
- 备份并让当前 Codex 重建 presence 文件。
- 所有 JSON 使用 `[Text.UTF8Encoding]::new($false)` 写入。
- 校验 PID 存活、协议 v2、路径存在，且文件前三字节不是 `EF BB BF`。

### 阶段 8：排除网络错误和假验收

`stream disconnected before completion ... /backend-api/codex/responses` 是 Codex 响应流网络中断，不等于 Chrome 插件故障。用 PowerShell `Start-Process` 打开 Edge 也只能证明系统能启动浏览器，不等于插件可以读取、点击或输入网页。

### 阶段 9：定位 WindowsApps EFS 与内容哈希 runtime

最终从 Desktop 日志和打包资源确认：

```text
bundled_executable_relocation_failed
copy_directory / copy_file
UNKNOWN errno=-4094
missingHelperPath
missingTransportModulePath
node-repl-missing
```

WindowsApps 源文件带有 `Encrypted` 属性。Node 的普通复制把 EFS 属性带到用户目录时失败；现有 fallback 只识别部分 Windows 错误码，未覆盖 `errno=-4094`。同时 Desktop 目标不是展示版本目录，而是由文件内容计算的哈希目录。

处理顺序：

1. `scripts/prewarm-codex-windowsapps-runtime.ps1` 动态计算内容哈希。
2. 建立 `%LOCALAPPDATA%\OpenAI\Codex\bin\<content-hash>`。
3. 建立 `%LOCALAPPDATA%\OpenAI\Codex\runtimes\cua_node\<content-hash>`。
4. 验证 `codex-computer-use.exe` 和 `helper_transport.js` 存在。
5. `scripts/repair-codex-bundled-marketplace-efs.ps1` 用 EFS-safe copy 建立当前版本镜像。
6. 验证 marketplace 全量文件大小和 SHA-256 一致，并清理过期 staging 目录。

### 阶段 10：Codex 更新后的版本漂移

Codex 更新后，旧镜像仍指向前一 MSIX 版本。重新运行 prewarm/inspect 脚本即可为新版本生成镜像；不需要重新改注册表、重装插件或修改 WindowsApps 权限。

### 阶段 11：最终重启与动态验收

后台脚本无法在终止 Codex 后继续可靠重启本任务，因此由用户从托盘完全退出，再手动启动 Codex。新任务中完成：

```text
Chrome：官方 browser-client 初始化并读取活动标签页标题和 URL
Computer Use：官方 computer-use-client 初始化并执行 sky.list_apps()
```

这才是最终通过；进程存在、pipe 存在、工具名称出现或系统命令打开浏览器都不是充分验收。

## 3. 涉及的本机文件与作用

| 文件或目录 | 发现的问题 | 处理方式 |
|---|---|---|
| `%LOCALAPPDATA%\OpenAI\extension\com.openai.codexextension.json` | manifest 缺失、路径旧或带 BOM | 备份后按当前插件路径重建，UTF-8 无 BOM |
| `HKCU\Software\Google\Chrome\NativeMessagingHosts\com.openai.codexextension` | 未注册或指向旧 manifest | 指向当前用户 manifest |
| `%USERPROFILE%\.codex\.tmp\bundled-marketplaces\openai-bundled` | marketplace 半残、staging 残留或版本旧 | 从当前 MSIX/解密镜像刷新 |
| `%USERPROFILE%\.codex\plugins\cache\openai-bundled` | Chrome/Computer Use 缓存版本旧，`latest` 错指 | 安装当前版本并重建 junction |
| `%USERPROFILE%\.codex\plugins\.plugin-appserver` | app-server 二进制来源或哈希不匹配 | 只使用当前 MSIX 匹配文件 |
| `%LOCALAPPDATA%\OpenAI\Codex\chrome-native-hosts-v2.json` | 死亡 PID、旧字段、BOM | 备份、重建、校验 v2/PID/无 BOM |
| `%USERPROFILE%\.codex\config.toml` | 固定 `notify`、不支持的 `deferred_executor=true`、外部 MCP workaround | 备份后删除对应覆盖；保留其他配置 |
| 用户环境变量 `CODEX_CLI_PATH` | 指向更新前的固定 CLI | 删除；不再固定 Desktop CLI |
| `%LOCALAPPDATA%\OpenAI\Codex\bin` | 曾混入 npm CLI；缺少内容哈希目标 | 对照 MSIX SHA-256，建立内容哈希目录 |
| `%LOCALAPPDATA%\OpenAI\Codex\runtimes\cua_node` | 只有 manifest tag 目录，缺少 Desktop 预期内容哈希目录 | 建立内容哈希 runtime 并验证 helper/transport |
| `%LOCALAPPDATA%\OpenAI\Codex\decrypted-bundled-resources\<MSIX版本>` | 原先不存在 | 建立 EFS-safe 镜像 |
| `CODEX_ELECTRON_BUNDLED_PLUGINS_RESOURCES_PATH` | 原先未指向可复制资源 | 指向当前版本解密镜像；更新后重新生成 |
| `%USERPROFILE%\.codex\node_repl\active_execs` | 遗留执行记录可能绑定已死内核 | 完全退出后只清理对应 JSON |

所有修改前都应备份；不要接管 WindowsApps ACL，不要删除整个 `.codex`，不要删除浏览器用户数据。

## 4. 本仓库脚本角色

| 脚本 | 角色 | 当前建议 |
|---|---|---|
| `check-codex-runtime.ps1` | 静态检查版本、路径、哈希和注册层 | 第一步运行 |
| `repair-codex-runtime-skew.ps1` | 早期多层修复器 | 仅在其 dry-run 结论与当前日志一致时使用 |
| `repair-chrome-app-server-registry.ps1` | 检查 app-server presence 与 BOM | 只处理 `No compatible...` / JSON 解析层 |
| `repair-official-node-repl-runtime.ps1` | 清理外部 MCP、旧进程和固定覆盖 | 中间清理工具，不等于动态验收 |
| `prewarm-codex-windowsapps-runtime.ps1` | 计算内容哈希并预热 Codex/CUA runtime | 本次最终关键修复之一 |
| `repair-codex-bundled-marketplace-efs.ps1` | 建立当前 MSIX 的 EFS-safe bundled 镜像 | 本次最终关键修复之一 |

`repair-chrome-native-lifecycle.ps1` 和延时 launcher 属于中间诊断阶段，不是最终根因修复。不要在未核对当前协议字段时手工伪造 app-server presence。

## 5. 不要重复的无效或高风险动作

```text
反复卸载/安装 Chrome 和 Computer Use 插件
反复切换“任意应用”授权开关
把普通 stdio node_repl MCP 当成官方可信桥
固定 CODEX_CLI_PATH 到某次版本目录
把 npm codex.exe 复制到 Desktop bin
跨版本复制旧 helper 或旧 node_modules
手工伪造 app-server PID/协议字段
只在旧任务中测试工具是否出现
用 Start-Process 打开浏览器后宣称插件验收通过
仅凭 nodeRepl.nativePipe 是否存在判定 Computer Use
修改 WindowsApps 所有权、ACL 或删除整个 .codex
```

## 6. 复发时的最短处理顺序

```powershell

.\scripts\prewarm-codex-windowsapps-runtime.ps1 -InspectOnly

# 2. 若报告 hash runtime 或 mirror 缺失，完全退出 Codex 后修复
.\scripts\prewarm-codex-windowsapps-runtime.ps1

# 3. 再次只读确认
.\scripts\prewarm-codex-windowsapps-runtime.ps1 -InspectOnly
```

随后手动启动 Codex 并新建两个任务：

```text
@Chrome 仅使用 Chrome 插件，读取当前活动标签页的标题和 URL；禁止使用系统命令替代。

```

## 7. 与 openai/codex#30365 的关系

本案例历史故障高度适用于 [openai/codex#30365](https://github.com/openai/codex/issues/30365)：都出现官方可信工具未注入、Sky runtime unavailable、helper/native pipe 缺失。不同之处是，本机后来确认了更具体的 WindowsApps EFS 复制失败和内容哈希 runtime 缺失，并已经通过预热与解密镜像恢复。

最合适的告知方式是在 #30365 下追加一条 follow-up comment，而不是创建重复 Issue。评论应包含：

- 当前 Codex、Chrome/Computer Use 插件版本。
- `bundled_executable_relocation_failed`、`UNKNOWN errno=-4094`、`missingHelperPath`、`missingTransportModulePath` 的脱敏日志。
- 内容哈希 runtime 和 EFS-safe mirror 的修复方式。
- 修复前失败与修复后 `sky.list_apps()` 成功的对照。
- 说明新版本中 `nodeRepl.nativePipe` 直接属性不是唯一验收标准。
- 本仓库对应文档和提交链接。

建议评论文本：

```markdown
Follow-up from Windows after reproducing the same trusted Computer Use injection failure.

Current verified versions: Codex Desktop `26.715.7063.0`; Chrome and Computer Use bundled plugins `26.715.52143`.

The original symptoms matched this issue: the official Computer Use wrapper reported that the Windows Sky runtime was unavailable, while the expected helper/pipe was missing. I later found a more specific local failure in the Desktop logs:

- `bundled_executable_relocation_failed`
- `UNKNOWN errno=-4094` while copying bundled resources from WindowsApps
- `missingHelperPath` / `missingTransportModulePath`

The WindowsApps sources were EFS-encrypted, and the Desktop content-hash runtime directories had not been materialized. I recovered the current MSIX files into SHA-256-verified content-hash targets, created a decrypted bundled-resources mirror with `CopyFileExW(COPY_FILE_ALLOW_DECRYPTED_DESTINATION)`, restarted Codex, and created a fresh task. The official Computer Use client then successfully executed `sky.list_apps()`.

One compatibility note: on the current build, `nodeRepl.nativePipe` can still be `undefined` while the official wrapper transport works. The real acceptance test was a successful official `sky.list_apps()` call, not the direct property alone.

Repair timeline and scripts: https://github.com/nayunzhishui/bug-of-codex-computer-chrome/blob/agent/document-runtime-skew-and-approval-bridge/docs/09-2026-07-20-%E5%AE%8C%E6%95%B4%E4%BF%AE%E5%A4%8D%E6%97%B6%E9%97%B4%E7%BA%BF%E4%B8%8E%E6%96%87%E4%BB%B6%E6%98%A0%E5%B0%84.md
```
